# Аудит Telegram-бота стоматологии

## Что уже хорошо
- Есть разделение по ролям (пациент/админ) через middleware.
- Реализована базовая логика записи, отмены и блокировки дней.
- Есть задел на расписание и автогенерацию слотов.
- Добавлен еженедельный бэкап БД.

## Ключевые зоны улучшений

### 1) Надёжность и безопасность
1. **Валидация конфигурации при старте**
   - Сейчас приложение не проверяет обязательность `BOT_TOKEN` и корректность `ADMIN_IDS` централизованно.
   - Рекомендация: фейлить старт с понятным логом, если токен пустой.

2. **Логирование вместо `print`**
   - В проекте смешаны `logging` и `print`.
   - Рекомендация: перейти на единый structured logging, добавить уровни `INFO/WARNING/ERROR` и контекст (`user_id`, `slot_id`).

3. **Гонки данных при бронировании**
   - Транзакция есть, но нет жёсткой гарантии уникальной активной записи на пациента на уровне БД.
   - Рекомендация: добавить индекс/ограничение (или отдельную таблицу active_appointments), чтобы бизнес-правила гарантировались БД.

4. **Резервные копии**
   - Бэкап отправляется только одному админу (`ADMIN_IDS[0]`).
   - Рекомендация: отправлять всем администраторам, добавить ручную кнопку «Сделать бэкап сейчас», добавить ротацию локальных копий.

### 2) UX для пациента
1. **Формат даты и локализация**
   - Сейчас пользователю показывается `YYYY-MM-DD`.
   - Рекомендация: показывать «Пн, 12 мая» и локальное время.

2. **Избыточные шаги в записи**
   - После выбора даты/времени не хватает экрана подтверждения «Вы точно хотите записаться?».
   - Рекомендация: добавить финальный confirm step с кнопками `Подтвердить / Изменить`.

3. **Отмена записи**
   - Жёсткое правило 24 часа полезно, но лучше предложить альтернативу.
   - Рекомендация: при запрете отмены выводить кнопку «Связаться с администратором».

4. **Список «Мои записи»**
   - Сейчас каждая запись отправляется отдельным сообщением.
   - Рекомендация: единый список + пагинация + единый callback для действий.

### 3) UX для администратора
1. **Настройки расписания через глобальный dict**
   - Используется `user_settings = {}` без FSM/TTL.
   - Рекомендация: перевести сценарий на FSM, чтобы избежать конфликтов между администраторами.

2. **Отмена дня без автоуведомлений**
   - В интерфейсе есть предупреждение, но нет автоматических уведомлений пациентам.
   - Рекомендация: массово уведомлять затронутых пациентов и собирать отчёт по доставке.

3. **Управление слотами**
   - Нет ручного управления отдельными слотами (перенос, принудительная отмена).
   - Рекомендация: добавить «Расписание на дату» с действиями по каждому слоту.

### 4) Архитектура и поддерживаемость
1. **Разделить слой БД и бизнес-логику**
   - В `database.py` сосредоточено очень много логики.
   - Рекомендация: вынести в репозитории/сервисы (`repositories/`, `services/`).

2. **Единая модель статусов**
   - Статусы слотов строковые (`free/booked/blocked/completed`) и размазаны по коду.
   - Рекомендация: enum-константы в одном месте + валидация переходов статусов.

3. **Миграции БД**
   - Сейчас схема меняется через `CREATE TABLE IF NOT EXISTS`.
   - Рекомендация: подключить Alembic/простой migration runner с версионированием.

### 5) Фичи, которые дадут заметный рост пользы
1. **Лист ожидания**
   - Если всё занято, пользователь может подписаться на освободившееся окно.

2. **Подтверждение записи за N часов**
   - Автонапоминание с кнопкой «Подтверждаю визит».

3. **Повторные визиты / шаблоны процедур**
   - Быстрая запись на «профгигиену через 6 месяцев».

4. **Мини-CRM метки**
   - Поле «комментарий к пациенту», история визитов, источник обращения.

5. **Метрики**
   - Конверсия: старт записи -> завершение.
   - Доля отмен.
   - Заполняемость по дням недели и часам.

## Приоритетный roadmap (быстрые победы)
1. Перевести все `print` в `logging`, добавить базовые error-алерты.
2. Доделать scheduler напоминаний (`scheduler/reminders.py`) и включить его в `bot.py`.
3. Сделать автоуведомление пациентов при блокировке дня.
4. Перевести админ-настройку расписания на FSM.
5. Добавить «лист ожидания» как next big feature.
